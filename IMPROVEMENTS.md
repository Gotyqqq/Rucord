# Анализ проекта Rucord и предложения по улучшению

Проект — аналог Discord: серверы, каналы, чат, голос, роли, DM. Ниже — рекомендации по архитектуре, коду и дизайну.

---

## 1. Архитектура и код

### 1.1 Роутинг

**Сейчас:** Нет React Router, «страницы» переключаются через `authPage` и наличие `user` в `App.jsx`. Нет URL для каналов/серверов (нельзя поделиться ссылкой, обновить страницу без потери контекста).

**Рекомендации:**
- Ввести **React Router** и URL вида:
  - `/` — лендинг
  - `/login`, `/register` — авторизация
  - `/channels/:serverId/:channelId?` — основной интерфейс (сервер + канал)
  - `/dm` — личные сообщения
- При смене сервера/канала обновлять `history`, при загрузке приложения — восстанавливать выбор из URL.
- Так можно будет открывать канал по ссылке и нормально работать с «Назад» в браузере.

### 1.2 Состояние приложения

**Сейчас:** Почти всё состояние живёт в `App.jsx` (десятки `useState`), много пропсов вниз по дереву, часть логики (socket, загрузка) смешана с UI.

**Рекомендации:**
- Вынести доменную логику в **контексты** или **Zustand/Redux**:
  - `ServersContext` — серверы, выбранный сервер, загрузка.
  - `ChannelsContext` — каналы, выбранный канал, сообщения, типинг, slowmode.
  - `VoiceContext` — текущий голосовой канал, участники, speaking.
  - Оставить в `App` только композицию и глобальные модалки (при необходимости).
- Подписки на Socket.IO лучше держать в одном месте (например, хук или контекст), а не в `useEffect` в `App.jsx`.
- Это упростит тесты и повторное использование логики.

### 1.3 Разделение App.jsx

**Сейчас:** Один файл ~540 строк: загрузка данных, сокеты, обработчики, разметка.

**Рекомендации:**
- Разбить на:
  - `App.jsx` — роуты, провайдеры, общий layout.
  - `MainLayout.jsx` — серверы + каналы + чат + участники + голос (без логики загрузки).
  - Кастомные хуки: `useServers()`, `useChannels()`, `useMessages()`, `useVoice()`, `useSocketEvents()` — загрузка и подписки на сокет.
- Модалки (CreateServer, JoinServer, Invite, Settings и т.д.) можно открывать через контекст или глобальное состояние (например, `modal: { type, props }`), чтобы не плодить булевы флаги в `App`.

### 1.4 Обработка ошибок и загрузка

**Сейчас:** Много `try/catch` с `console.error` или `alert`, нет единого UX для ошибок сети/сервера.

**Рекомендации:**
- Ввести слой API (axios/fetch) с перехватчиками: при 401 — редирект на логин, при 5xx — тост «Ошибка сервера, попробуйте позже».
- Показывать тосты/снэкбары вместо `alert()` для ошибок (уже есть `notification-toast` — можно обобщить под «ошибка»).
- Для критичных действий (создание сервера, отправка сообщения) — индикаторы загрузки (disabled кнопка + спиннер), чтобы не было двойных отправок.

### 1.5 Тесты

**Сейчас:** Тестов нет.

**Рекомендации:**
- Добавить **Vitest** + **React Testing Library** во фронтенд.
- Сначала покрыть: утилиты (`api`, `socket`), контекст авторизации, ключевые хуки.
- Затем — снимки или простые тесты для страниц входа/лендинга и главного layout (без сокетов).
- На бэкенде — тесты для маршрутов (auth, servers, channels, messages) и, по возможности, сокет-событий.

### 1.6 Типизация

**Сейчас:** Весь фронт на JS (JSX).

**Рекомендации:**
- Постепенный переход на **TypeScript**: начать с `utils/`, `context/`, типов для API (модели Server, Channel, Message, User). Это уменьшит количество багов при рефакторинге и улучшит подсказки в IDE.

## 2. Дизайн и UX

### 2.1 Стили и темизация

**Сейчас:** Один большой `App.css`, все цвета захардкожены (#36393f, #2f3136, #5865f2 и т.д.), шрифт body — Segoe UI, в HTML подключается Nunito (используется не везде).

**Рекомендации:**
- Ввести **CSS-переменные** для темы в `:root`:
  - `--bg-primary`, `--bg-secondary`, `--bg-tertiary`
  - `--text-primary`, `--text-muted`, `--text-link`
  - `--accent`, `--accent-hover`, `--danger`, `--success`
  - `--radius-sm`, `--radius-md`, `--shadow-modal`
- Один файл (например, `theme.css`) задаёт значения для тёмной темы; при желании можно добавить светлую тему, переопределяя переменные.
- Унифицировать шрифты: либо везде Nunito, либо везде одна система (например, Segoe UI + один акцентный). Убрать дублирование в `index.html` и `App.css`.

### 2.2 Дизайн-система (компоненты)

**Сейчас:** Нет переиспользуемых примитивов, кнопки/инпуты/карточки стилизованы разными классами.

**Рекомендации:**
- Выделить базовые компоненты в `src/components/ui/`:
  - **Button** (primary, secondary, danger, ghost, icon), **Input**, **Select**, **Modal** (обёртка с затемнением и закрытием по Escape).
  - **Avatar** (инициалы/картинка), **Badge** (счётчик, статус).
  - **Toast** — единый компонент уведомлений (успех, ошибка, информация).
- Использовать их в существующих экранах и модалках — так интерфейс станет визуально единее и проще поддерживать.

### 2.3 Лендинг и первые экраны

**Сейчас:** Лендинг простой: логотип, заголовок, два CTA, блок «Как это устроено», футер. Стили есть в `App.css` (`.landing-*`).

**Рекомендации:**
- Добавить лёгкую анимацию появления секций (например, `opacity` + `transform` при скролле или при монтировании).
- Увеличить контраст CTA-кнопок: чёткий primary (например, `#5865f2`) и вторичная с обводкой.
- На мобильных — бургер-меню для «Войти» / «Регистрация», если навигация переносится.
- Небольшой визуальный акцент: градиент или паттерн на фоне hero-секции, не перегружая страницу.

### 2.4 Чат и список сообщений

**Сейчас:** Сообщения в виде списка, есть hover, аватарки, реакции, вложения. Стили — `.message`, `.message-body` и др.

**Рекомендации:**
- **Группировка по автору:** если подряд идут сообщения одного пользователя за короткий промежуток (например, 5–10 минут), скрывать повторный аватар и имя, оставлять только «хвост» сообщения — как в Discord. Это сократит визуальный шум.
- **Визуальное разделение по времени:** разделители «Сегодня», «Вчера», «В понедельник» и т.д. между блоками сообщений.
- **Плавная подгрузка вверх:** при скролле вверх подгружать старые сообщения без рывка (например, сохранять позицию скролла после вставки новых элементов).
- **Фокус и доступность:** видимый `:focus-visible` у сообщений и кнопок (реакции, меню), чтобы навигация с клавиатуры была понятной.

### 2.5 Список серверов и каналов

**Сейчас:** Слева — иконки серверов (72px), справа — список каналов с изменяемой шириной. Есть hover-эффекты.

**Рекомендации:**
- **Тултипы:** при наведении на иконку сервера показывать название сервера во всплывающей подсказке (если ещё нет), чтобы при большом количестве серверов было проще ориентироваться.
- **Активное состояние:** чётко выделять выбранный сервер и выбранный канал (уже есть `.active` — проверить контраст на фоне).
- **Скролл:** если каналов много — видимая полоса прокрутки или «затенение» сверху/снизу, чтобы было понятно, что список прокручивается.
- **Ресайзер:** кроме изменения курсора при наведении на разделитель, можно добавить тонкую полоску подсветки при перетаскивании (уже есть hover — можно усилить при `drag`).

### 2.6 Модальные окна и overlay

**Сейчас:** Много модалок (создание сервера, приглашение, настройки и т.д.), стили разбросаны по `App.css`.

**Рекомендации:**
- Единый базовый компонент **Modal**: полупрозрачный overlay, блок по центру, закрытие по клику вне и по Escape, блокировка скролла body, фокус в ловушке (tab внутри модалки).
- Все существующие модалки обернуть в этот компонент — уменьшится дублирование и улучшится доступность.
- Анимация: лёгкое появление overlay (fade) и модального окна (scale/fade) за 150–200 ms.

### 2.7 Уведомления (тосты)

**Сейчас:** Есть `.notification-toast` для упоминаний и ошибок; клик по тосту ведёт в канал.

**Рекомендации:**
- Один компонент **Toast** с типами: `mention`, `error`, `info`, `success`. Иконка и цвет в зависимости от типа.
- Позиция: например, правый нижний угол, стек тостов сверху вниз; авто-скрытие через 4–5 сек с возможностью закрыть по кнопке.
- Звук (опционально) при упоминании — настраиваемый в настройках пользователя.

### 2.8 Адаптивность и мобильные

**Сейчас:** Судя по структуре, интерфейс рассчитан на десктоп (три колонки: серверы | каналы+чат | участники).

**Рекомендации:**
- На узких экранах (< 768px):
  - Скрывать список каналов и показывать только чат; открытие списка каналов — по кнопке (иконка «каналы» или гамбургер).
  - Список участников — выдвижная панель справа или снизу по кнопке.
  - Кнопка «Пригласить» — в шапке чата или в меню.
- Использовать `max-width` и медиа-запросы для панелей; при необходимости — отдельные компоненты-обёртки для мобильного layout (например, `ChannelListDrawer`).

### 2.9 Доступность (a11y)

**Сейчас:** Есть единичные `aria-*` и `role` (например, у ресайзера), но системной работы с a11y нет.

**Рекомендации:**
- Ключевые интерактивные элементы: кнопки, ссылки, поля ввода — с видимым `:focus-visible` (outline или box-shadow).
- Модалки: `aria-modal="true"`, `aria-labelledby` на заголовок, фокус при открытии на первый фокусный элемент, возврат фокуса при закрытии.
- Список сообщений: семантика списка (`ul`/`li` или `role="list"`), у каждого сообщения — заголовок с именем и датой для скринридеров.
- Избегать только визуальной индикации (например, «онлайн» не только цветом, но и текстом/иконкой «в сети»).

### 2.10 Микро-анимации

**Сейчас:** В основном `transition` для hover (цвет, фон). Мало обратной связи при действиях.

**Рекомендации:**
- Отправка сообщения: лёгкая анимация появления нового сообщения (fade-in или slide-in снизу).
- Реакции: при добавлении эмодзи — краткая анимация (scale или «прыжок»).
- Загрузка: скелетоны для списка сообщений или списка каналов вместо пустого экрана.
- Переключение каналов: индикатор загрузки в области чата, пока подгружаются сообщения.

## 3. Бэкенд и инфраструктура

### 3.1 База данных

**Сейчас:** В коде используется sql.js (WASM); в README упоминается better-sqlite3 — стоит привести документацию в соответствие с фактическим движком.

**Рекомендации:**
- В README и конфигах явно указать: SQLite через sql.js (или better-sqlite3, если переключитесь).
- При росте нагрузки рассмотреть миграцию на PostgreSQL и миграции схемы (например, node-pg-migrate или аналог).

### 3.2 Безопасность

**Сейчас:** JWT, bcrypt, есть `SECURITY.md` и мастер-аккаунты.

**Рекомендации:**
- Rate limiting на критичные маршруты (логин, регистрация, создание инвайтов).
- Валидация и санитизация входных данных (длина имени сервера/канала, контент сообщений).
- Загрузка файлов: проверка MIME-типа и размера, хранение вне корня приложения, отдача по безопасному пути.

### 3.3 Переменные окружения фронтенда

**Сейчас:** API и сокеты идут через proxy на localhost:3001; для продакшена нужен другой хост.

**Рекомендации:**
- Ввести `VITE_API_URL` и `VITE_WS_URL` (или один `VITE_API_ORIGIN`) в `.env` и использовать в `api.js` и `socket.js`. В dev — по умолчанию пусто (proxy), в production — подставлять реальный хост. Так один билд не будет привязан к localhost.
