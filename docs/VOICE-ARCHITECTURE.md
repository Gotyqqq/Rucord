# Архитектура голосовой связи

## Текущая схема (P2P через WebRTC)

```
[Участник A]  <--->  [Сервер]  <--->  [Участник B]
     |                     |                    |
     |   offer/answer/ICE   |   (только сигналы) |
     +=====================+====================+
     |                     |                    |
     +-------- Прямое UDP (RTP) между A и B ----+
```

- **Сервер** только пересылает сигналы (offer, answer, ICE) по Socket.IO в комнату `user_<toUserId>`.
- **Медиа** идут напрямую между браузерами (UDP, порты могут быть любыми).

### Почему в РФ P2P часто не работает

1. **NAT и файрволы** — у многих провайдеров жёсткий NAT или блокировка нестандартных UDP-портов.
2. **Блокировки** — часть трафика к STUN (например, к Google) или сам WebRTC могут резать или дросселировать.
3. **Нет TURN** — при неудаче прямого соединения нужен TURN-сервер (трафик через сервер). Сейчас только STUN, своего TURN нет.
4. **Разные сети** — корпоративные/мобильные/разные провайдеры часто не могут установить прямое UDP-соединение.

В итоге: **сигнализация** (Socket.IO) доходит, **speaking** и индикаторы работают, а **аудиопоток по WebRTC** — нет, звука не слышно.

## Альтернатива: релей через сервер (WebSocket)

Трафик голоса идёт **только через ваш сервер** по тому же Socket.IO (или WebSocket), без P2P.

```
[Участник A]  ----voice_audio---->  [Сервер]  ----voice_audio---->  [Участник B]
     |                                    |                              |
  Микрофон → Opus → Socket.IO         broadcast                  Socket.IO → decode → динамик
```

- Каждый клиент **кодирует** микрофон (Opus в WebM), шлёт чанки на сервер событием `voice_audio`.
- Сервер **разбрасывает** эти чанки всем в комнате `voice_<channelId>`, кроме отправителя.
- Остальные клиенты **принимают** чанки, декодируют и **воспроизводят** через Web Audio API.

Плюсы:
- Один тип соединения (Socket.IO), те же порты и правила, что и для чата.
- Не зависит от UDP/NAT/STUN/TURN и ограничений на WebRTC в сети.
- Проще отлаживать: всё идёт через ваш бэкенд.

Минусы:
- Нагрузка на сервер и канал (весь голосовой трафик через него).
- Задержка чуть выше, чем при идеальном P2P (обычно приемлемо для голоса).

## Режимы в коде

В `frontend/src/utils/voiceConfig.js` задаётся флаг **`VOICE_USE_RELAY`** (по умолчанию `true`):

- **Relay (по умолчанию):** голос идёт через сервер по Socket.IO (`voice_audio`). Не зависит от NAT/WebRTC, стабильнее в РФ.
- **P2P** (`VOICE_USE_RELAY = false`): WebRTC между участниками, сервер только пересылает сигналы (offer/answer/ICE). Может не работать за жёстким NAT или при блокировках.

Переключение: в `voiceConfig.js` изменить `VOICE_USE_RELAY` на `false` и пересобрать фронт.

## Релей: формат и ограничения

- **Отправка:** MediaRecorder (audio/webm; codecs=opus), чанки по ~60 ms, отправка через `voice_audio` (binary).
- **Приём:** `decodeAudioData` → очередь по пользователю → воспроизведение через `AudioContext` (без `<audio>` и автовоспроизведения).
- **Мьют/дефен:** при мьюте не шлём чанки; при дефене не воспроизводим входящие (как и сейчас).

- **Бэкенд:** в `backend/socket/chat.js` — обработчик `voice_audio`: проверка комнаты `voice_<channelId>`, мьют, лимит размера чанка (64 KB), рассылка в `socket.to(voice_<channelId>)` с `userId` и `data`.
- **Фронт:** в `VoicePanel.jsx` при `VOICE_USE_RELAY`: отправка — MediaRecorder (audio/webm;codecs=opus, 60 ms) с `processedStreamRef`, emit `voice_audio`; приём — подписка на `voice_audio`, decodeAudioData, очередь по userId, воспроизведение через AudioContext. Мьют/дефен и баннер «Включить звук» работают так же.
